# EF Core ‚Äî ExecuteDelete / ExecuteUpdate with ToTable + ToView

A minimal reproduction and fix for an EF Core bug where `ExecuteDelete` and
`ExecuteUpdate` target a **view** instead of the underlying physical table when
an entity is dual-mapped with `ToTable("‚Ä¶").ToView("‚Ä¶")`.

## The pattern

```csharp
modelBuilder.Entity<EDIMessage>(eb =>
{
    eb.ToTable("EDIMessage")       // INSERT / UPDATE / DELETE (tracked) ‚Üí physical table
      .ToView("kvw_EDIMessage");   // SELECT                             ‚Üí view
});
```

`kvw_EDIMessage` is a `UNION ALL` view over `EDIMessage` (active) and
`EDIMessage_Archive` (archived), giving callers a single read surface across
both tables while keeping writes isolated to the active table.

## The bug (EF Core 7 ‚Äì 9)

EF Core's bulk-DML operations (`ExecuteDelete` / `ExecuteUpdate`) follow the
IQueryable's source ‚Äî which is the view ‚Äî and generate SQL targeting it:

```sql
-- generated by EF Core 7 / 8 / 9 (wrong)
DELETE FROM "kvw_EDIMessage" AS "k" WHERE "k"."Status" = 'Processed'
UPDATE "kvw_EDIMessage" AS "k" SET "Status" = @p WHERE "k"."Status" = 'Pending'
```

Databases reject DML on `UNION ALL` views. Tracked `SaveChanges` works
correctly (it uses the table), but the bulk operations fail.

**Affected versions:** EF Core 7.0, 8.0, 9.0 (confirmed by IL inspection).  
**Fixed in:** EF Core 10.0 (natively ‚Äî no workaround needed).

## Workaround for EF Core 7 / 8 / 9

Two files fix the routing at the expression-tree level with no changes to
call sites, no new `DbSet`, and no SQL string manipulation.

### How it works

EF Core exposes two `virtual` hooks on
`RelationalQueryableMethodTranslatingExpressionVisitor` that output the
DELETE / UPDATE target table:

```csharp
protected virtual bool IsValidSelectExpressionForExecuteDelete(
    SelectExpression select, EntityShaperExpression shaper,   // EF7
    out TableExpression tableExpression);

protected virtual bool IsValidSelectExpressionForExecuteUpdate(
    SelectExpression select, EntityShaperExpression shaper,   // EF7
    out TableExpression tableExpression);
```

`ViewToTableRedirectVisitor` overrides both. When the base resolves the target
to the view, the override swaps it for the entity's physical table:

1. Looks up the entity's physical table via `entityType.GetTableMappings()`
2. Creates a new `TableExpression` for the physical table (internal constructor
   accessed via reflection ‚Äî no public one exists)
3. Copies the alias from the original view `TableExpression` so `WHERE` column
   references stay valid
4. Replaces the matching entry in `SelectExpression._tables` so the
   reference-equality check inside `QuerySqlGenerator.VisitDelete/VisitUpdate`
   passes

Result:

```sql
-- generated after the fix (correct)
DELETE FROM "EDIMessage" AS "k" WHERE "k"."Status" = 'Processed'
UPDATE "EDIMessage" AS "k" SET "Status" = @p WHERE "k"."Status" = 'Pending'
```

### Registration

```csharp
optionsBuilder
    .UseSqlServer(connectionString)   // or UseSqlite / any provider
    .ReplaceService<
        IQueryableMethodTranslatingExpressionVisitorFactory,
        ViewToTableRedirectVisitorFactory>();
```

One line alongside your existing provider call. No other changes needed.

### API differences across EF Core versions

The visitor's override signatures differ between versions. If you adapt this
workaround for EF Core 8 or 9, update the shaper parameter type:

| EF Core | `IsValidSelectExpressionForExecuteDelete` | `IsValidSelectExpressionForExecuteUpdate` |
|---------|------------------------------------------|------------------------------------------|
| 7.x | `EntityShaperExpression shaper` | `EntityShaperExpression shaper` |
| 8.x ‚Äì 9.x | `StructuralTypeShaperExpression shaper` | `TableExpressionBase targetTable` |
| 10.x | *(single-arg overload ‚Äî fixed natively)* | `TableExpressionBase targetTable` |

## Upgrading to EF Core 10

Remove the workaround entirely:

1. Delete `ViewToTableRedirectVisitor.cs` and `ViewToTableRedirectVisitorFactory.cs`
2. Remove the `.ReplaceService<‚Ä¶>()` call from your `DbContextOptionsBuilder`

Nothing else needs to change. The `ToTable().ToView()` mapping and all call
sites remain identical.
